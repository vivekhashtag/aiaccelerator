## Why We Need Memory

A combinational circuit forgets instantly — its output depends only on its current inputs. But computing needs **history**: a counter must know its last count, a CPU must know which instruction is next. A **sequential** circuit's output depends on both current inputs *and* its stored past.

<Callout type="insight" title="Feedback is memory">

The trick that creates memory is **feedback** — looping a circuit's output back into its input. A loop can hold a stable value indefinitely, even after the inputs that set it go away. That self-sustaining loop is the physical basis of every bit ever stored.

</Callout>

---

## The SR Latch: The First Memory Element

The simplest memory cell is two cross-coupled NOR gates — the **SR (Set-Reset) latch**.

<SRLatchDiagram caption="Two NOR gates, each feeding the other. That feedback loop is what lets the latch hold a value after the inputs go quiet — the first taste of memory." />

- **S=1, R=0** → Q is **set** to 1.
- **S=0, R=1** → Q is **reset** to 0.
- **S=0, R=0** → Q **holds** its previous value. *This is memory.*
- **S=1, R=1** → **forbidden** (both outputs forced low, breaking Q = NOT Q̄).

<Callout type="pitfall" title="The latch is level-sensitive">

While S or R is active, the output reacts *immediately*. In a system where we want every change to happen together at one precise moment, that's a problem — glitches and races creep in. The fix is to add a clock, which gives us the flip-flop.

</Callout>

---

## The D Flip-Flop: Foundation of Synchronous Design

The **D (Data) flip-flop** eliminates the forbidden state and adds clock control. It is the single most important sequential element in digital design.

On the **rising edge** of the clock (0 → 1 transition), the flip-flop captures whatever is on its `D` input and holds it on `Q` until the next rising edge. Between edges, changes on `D` are ignored.

| D | Clock | Q (next) |
|---|-------|----------|
| 0 | ↑ edge | 0 |
| 1 | ↑ edge | 1 |
| X | no edge | Q (unchanged) |

<DFlipFlopTiming caption="Watch Q: it only changes at a rising clock edge (dashed lines), copying whatever D is at that instant. Between edges, D can wiggle all it likes — Q ignores it." />

<Callout type="why" title="One clock edge, one synchronised update">

In a synchronous system, **every** flip-flop updates on the *same* clock edge. Combinational logic computes new values during one clock period; then, all at once, the flip-flops latch them on the next edge. This turns a messy analog pile of gates into a perfectly predictable, step-by-step computation pipeline. Every register in a CPU, GPU, and FPGA is built from D flip-flops:

</Callout>

## JK and T Flip-Flops

The D flip-flop is the workhorse, but two close cousins round out the family:

- **JK flip-flop** — the SR latch's forbidden state, *fixed*. `J` sets, `K` resets — and the case that was illegal for SR (`J=K=1`) now does something useful: it **toggles** (Q flips to its opposite). It's the most general single-bit element.
- **T (Toggle) flip-flop** — a JK with its two inputs tied together. When `T=1` it flips on every clock edge; when `T=0` it holds. One input, one job.

| Type | Inputs | On the clock edge |
|---|---|---|
| **D** | D | Q ← D |
| **JK** | J, K | 00 hold · 01 reset · 10 set · 11 **toggle** |
| **T** | T | T=0 hold · T=1 **toggle** |

<Callout type="insight" title="Toggle is what makes counters count">

The T flip-flop's "flip on every tick" behaviour is the seed of *counting* — chain them and each stage toggles half as often as the one before it, producing a binary count. That's the next building block.

</Callout>

## Counters

A **counter** is a register that steps through a sequence of values. The simplest — a **ripple counter** — is just T flip-flops chained so each one's output clocks the next:

<RippleCounterDiagram caption="Three toggle flip-flops in a chain. Q0 flips every clock, Q1 every two, Q2 every four — so Q2 Q1 Q0 counts 000 → 111 in binary. This is a divide-by-2 chain." />

Because each bit toggles at half the rate of the one below, a counter is also a **frequency divider**: an N-bit counter divides the clock by 2ᴺ. Counters generate addresses (stepping through memory), keep time (a timer that counts out a fixed delay), and sequence operations — like walking a MAC engine across every element of a matrix.

<Callout type="pitfall" title="Ripple vs synchronous">

A ripple counter is cheap but its bits update one-after-another (the same rippling delay as a ripple-carry adder), so the count is briefly *wrong* mid-transition. **Synchronous counters** clock every flip-flop from the *same* edge — more logic, but all bits change at once. High-speed designs always use synchronous.

</Callout>

## Registers & Shift Registers

A **register** is just N D flip-flops sharing one clock — it stores a multi-bit value (an 8-bit register holds a byte; a 32-bit register holds an int or float). The accumulator in a MAC unit is a register, usually made *wider* than its inputs (32–48 bits) so many products can pile up without overflowing.

A **shift register** chains flip-flops so data marches one position per clock edge:

<ShiftRegisterDiagram caption="Four flip-flops in a chain. On every clock edge each one hands its value to the next, so data marches one step to the right — and each Q is a parallel tap." />

Shift registers are everywhere: serial↔parallel conversion (SPI, UART), pipeline delay buffers in DSP, pseudo-random generators (LFSRs), and the serializers in PCIe and DDR.

## Finite State Machines: Logic That Remembers

A **Finite State Machine (FSM)** is the general pattern for control logic. It has a finite set of **states** (stored in flip-flops), **inputs**, **outputs**, and rules for transitioning between states and producing outputs.

- **Moore machine** — outputs depend only on the *current state*.
- **Mealy machine** — outputs depend on current state **and** current inputs (reacts faster, often needs fewer states).

The design flow: draw a **state diagram** → **encode** states in binary (N states need ⌈log₂N⌉ flip-flops) → build a **state table** → derive **next-state and output logic** → implement with flip-flops + gates.

<FSMTrafficLight caption="A Moore machine: each state (RED/GREEN/YELLOW) drives a fixed output. T=1 advances to the next state; T=0 loops in place — 'stay until the timer expires.'" />

<Callout type="key" title="FSMs run the accelerators">

This exact pattern controls AI hardware. A MAC unit's controller is an FSM cycling through **IDLE → LOAD_WEIGHTS → LOAD_INPUT → COMPUTE → OUTPUT → IDLE**. CPU control units, memory controllers, protocol handlers, and accelerator sequencers are all FSMs. Combinational logic computes *values*; sequential logic and FSMs decide *what happens when*.

</Callout>
