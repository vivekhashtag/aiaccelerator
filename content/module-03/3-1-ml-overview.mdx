## What Is Being Computed With All Those MACs?

In Module 1 you built the MAC operation: `acc = acc + (A × B)`. In Module 2 you saw a GPU run tens of thousands of MACs in parallel, and the roofline model told you whether a workload is compute- or memory-bound. This module answers the question those two left open: **what, exactly, are all those MACs computing?**

The answer is a **model** — a function `f(x; θ)` with millions of tunable numbers (**weights**, `θ`) that maps inputs to outputs. Stripped of abstraction, deep learning is three sentences: (1) a huge number of MACs arranged into layers, (2) a function from inputs to outputs, (3) a learning algorithm that adjusts the weights by measuring error and propagating blame backward. This lesson sets up the vocabulary; the rest of the module unpacks each sentence.

---

## What "Learning" Means

Traditional programming is explicit — a human writes the rules. Machine learning **inverts** this: you provide examples, and an algorithm discovers the rules.

<Formula label="The definition of machine learning" caption="Find weights θ so the function approximates the data — and generalises to inputs it never saw.">find θ such that f(x; θ) ≈ y for all (x, y), and it generalises</Formula>

The weights `θ` are what is "learned" — just numbers in memory, adjusted until the function produces correct outputs. Everything else in this module is *how* we find good `θ`.

## The Landscape: Four Ways to Learn

Not all learning is the same. The difference is **where the supervision comes from**.

<MLLandscape caption="The four paradigms. Every one is still 'find θ so f(x;θ) ≈ y' — they differ only in what signal tells the model it's wrong." />

- **Supervised** — labeled `(input, output)` pairs. **Regression** predicts a continuous value (house price); **classification** predicts a category (cat/dog/spam). *Every neural network in this course is supervised.*
- **Unsupervised** — inputs only, no labels. **Clustering** groups similar examples (k-means); **dimensionality reduction** compresses data (PCA, autoencoders).
- **Self-supervised** — labels are generated *from the data itself*: predict the next word, or fill in a masked patch. This is how GPT, BERT, and CLIP are trained on oceans of unlabelled data — and it's what makes LLMs possible at scale.
- **Reinforcement** — an agent takes actions, receives rewards, and learns a policy that maximises cumulative reward. Powers game-playing, robotics, and **RLHF** (Module 4) — the alignment step behind modern chatbots.

<Callout type="why" title="This taxonomy is your routing table">

When a real problem lands on your desk, the first question is *which box is this?* It decides everything downstream — the data you need, the model class, the loss, and the hardware. Lesson 3.2 adds the *classical* models that often win the supervised/tabular box, and Module 4 builds out the self-supervised box (foundation models).

</Callout>

## The Supervised Framework: Three Splits

You never judge a model on the data it learned from. Split the data three ways, each with a strict job:

<DataSplitBar caption="Train (learn the weights) · Validation (tune architecture and hyperparameters) · Test (report final performance exactly once)." />

<Callout type="pitfall" title="The test set is sacred — touch it once">

The test set must **never** influence any decision: not the architecture, not the hyperparameters, not when to stop training. A model that hits 95% validation but 70% test has **overfit to the validation set** — the engineer made too many choices by peeking at it. The test score is honest only because it was used exactly once.

</Callout>

## Loss Functions: Quantifying "Wrong"

A **loss function** turns "how wrong was the prediction" into a single number to minimise. The choice depends on the task.

<Formula label="The two losses you'll use most" caption="MSE for regression; cross-entropy for classification.">MSE = (1/N) Σ (yᵢ − ŷᵢ)²          CE = −(1/N) Σ yᵢ·log(ŷᵢ)</Formula>

**MSE** squares the error: all-positive, penalises big misses hardest, clean derivative. **Cross-entropy** punishes *confident wrong* answers brutally — saying "99% sure it's A" when it's B costs `−log(0.01) = 4.6`, while honest uncertainty (50%) costs only `0.69`. That's what pushes a classifier toward **calibrated** confidence (we measure calibration in Lesson 3.7).

## The Bias–Variance Tradeoff

Every model can fail in two opposite ways:

- **High bias (underfitting)** — too simple to capture the pattern. Bad on *both* training and test data. *Fix:* more capacity, train longer, less regularisation.
- **High variance (overfitting)** — memorises the training data *including its noise*. Great on training, poor on test. *Fix:* more data, regularisation (dropout, weight decay), early stopping, less capacity.

<BiasVarianceCurve caption="As complexity grows, bias falls but variance rises. Total error is their sum — a U-curve. The best model sits at the bottom: complex enough to learn the pattern, simple enough to ignore the noise." />

<Callout type="insight" title="Modern deep learning bends the classic curve">

The U-curve is the classical picture. Very large, **overparameterised** networks (more parameters than training examples) show *double descent* — past a threshold, test error falls *again*, achieving low bias and low variance together. The classical theory doesn't predict this, and it's part of why scale works for LLMs. You'll feel both regimes in Lab 3-A.

</Callout>

## Gradient Descent: The Learning Algorithm

How do we actually *find* good weights? Picture the loss as a landscape — a valley whose floor is the best `θ`. **Gradient descent** repeatedly steps downhill: compute the gradient (steepest-ascent direction) and move the opposite way.

<Formula label="The gradient descent update" caption="Step against the gradient; α (the learning rate) sets the step size.">θ ← θ − α · ∇θ L(θ)</Formula>

<GradientDescentLandscape caption="Each step moves θ downhill against the gradient. Steps shrink near the minimum, where the slope flattens. The learning rate α controls how far each step goes — too big overshoots, too small crawls." />

How much data you use per step gives three flavours:

<StatGrid items={[
  { value: "Batch", label: "whole dataset / step", sub: "stable, slow" },
  { value: "SGD", label: "one sample / step", sub: "noisy, fast" },
  { value: "Mini-batch", label: "32–512 / step", sub: "the standard" }
]} caption="Mini-batch is the default: fast updates, manageable noise, and it maps to one matrix multiply per batch — the GPU sweet spot from Module 2." />

**Optimisers** make the steps smarter. **Momentum** accumulates velocity in consistent directions (a ball rolling downhill). **Adam** — the default — keeps a per-parameter learning rate by tracking running averages of the gradient and its square, so noisy or flat directions self-correct.

<Callout type="why" title="Adam's memory cost shapes what you can train">

Adam stores two extra vectors (`m` and `v`) the size of the model. For a 7B model in FP32 that's `7B × 2 × 4 bytes = 56 GB` of **optimiser state alone** — on top of weights and gradients. This is the real reason large models train only on big multi-GPU rigs even when the weights themselves would fit. It's the training-side echo of the memory-bandwidth story from Module 2, and we quantify it fully in Lesson 3.6.

</Callout>

## Recap

- A model is a function `f(x; θ)`; **learning** is finding weights `θ` so it fits the data *and generalises* — measured honestly only on held-out data.
- Four paradigms — **supervised, unsupervised, self-supervised, reinforcement** — differ only in where the supervision signal comes from; this taxonomy routes every real problem.
- **Train / validation / test** have strict jobs; the test set is used **once**.
- **Loss functions** (MSE, cross-entropy) define "wrong"; cross-entropy rewards calibrated confidence.
- The **bias–variance tradeoff** is the U-curve of generalisation — with a modern *double-descent* twist at scale.
- **Gradient descent** (mini-batch + Adam) is the loop that trains everything ahead — and its memory cost foreshadows the training-vs-inference gap of Lesson 3.6.
