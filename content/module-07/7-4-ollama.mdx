## What Ollama Adds

llama.cpp is powerful but low-level — you build it, find a GGUF file, and manage flags. **Ollama** wraps it in a clean, `apt`/`docker`-style experience: pull models by name, automatic GPU detection, and a built-in API server. Same engine, zero config:

## The CLI Workflow

Everything starts with pulling a model by name from the Ollama library, then running it:

```bash
ollama pull llama3.2          # download Llama-3.2-3B (Q4_K_M, ~2 GB)
ollama pull llama3.1:8b       # a specific size tag (~4.4 GB)
ollama run llama3.2           # interactive chat in the terminal
ollama list                   # show installed models
ollama rm llama3.2            # delete a model
```

**Step by step:**

1. `ollama pull <name>` downloads a ready-made **Q4_K_M GGUF** (the sweet-spot quant from 7.3) — no hunting for files.
2. The `:8b` / `:70b` **tag** picks the size variant of a model family.
3. `ollama run` drops you into a chat REPL; `list` and `rm` manage what's on disk, just like a package manager.
4. On first run Ollama auto-detects your GPU (CUDA, Apple Metal, or AMD ROCm) and offloads as many layers as fit — no flags needed.

## The REST and OpenAI APIs

Once Ollama is running, it serves an HTTP API on `localhost:11434`. The native endpoint streams tokens line by line:

```python
import requests, json

resp = requests.post("http://localhost:11434/api/generate", json={
    "model": "llama3.2",
    "prompt": "Explain PagedAttention in 3 sentences.",
    "stream": True,
    "options": {"temperature": 0.7, "num_ctx": 4096},
}, stream=True)

for line in resp.iter_lines():
    if line:
        chunk = json.loads(line)
        print(chunk.get("response", ""), end="", flush=True)
        if chunk.get("done"):
            break
```

**Step by step:**

1. You POST to `/api/generate` with the model name, prompt, and `options` (the decoding knobs — `temperature`, `num_ctx` context size).
2. `stream=True` makes Ollama send back one JSON object per token; you parse each line and print its `response` field as it arrives.
3. The loop stops when a chunk has `"done": true` — the end-of-generation signal.

Even simpler: Ollama is **OpenAI-compatible**, so the official SDK works by just changing the URL:

```python
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
stream = client.chat.completions.create(
    model="llama3.2",
    messages=[{"role": "user", "content": "What is vLLM?"}],
    stream=True,
)
for chunk in stream:
    print(chunk.choices[0].delta.content or "", end="", flush=True)
```

**Step by step:**

1. Same OpenAI client as vLLM in 7.2 — only the `base_url` (port 11434) and a dummy `api_key` change.
2. This means code written against OpenAI, vLLM, *or* Ollama is **interchangeable** — you can prototype locally on Ollama and deploy on vLLM without rewriting the app.

## Custom Models with Modelfiles

A **Modelfile** bakes a base model, system prompt, and default parameters into a named, reusable model — like a `Dockerfile` for an LLM persona:

```dockerfile
# Modelfile
FROM llama3.1:8b

SYSTEM """You are an expert AI infrastructure engineer.
You give precise, technical answers with code examples.
You know vLLM, llama.cpp, TensorRT, and quantization deeply."""

PARAMETER temperature 0.3      # low = more deterministic (technical content)
PARAMETER num_ctx 8192         # large context for long technical discussions
PARAMETER top_p 0.9
```

```bash
ollama create ai-infra-expert -f Modelfile   # build the custom model
ollama run ai-infra-expert                    # use it like any other model
```

**Step by step:**

1. `FROM` picks the base model to build on.
2. `SYSTEM` hard-codes a system prompt so you don't resend it with every request.
3. `PARAMETER` lines set defaults — here a low `temperature` for deterministic technical answers and a big `num_ctx`.
4. `ollama create -f Modelfile` compiles it into a named model; from then on it behaves like a built-in, with the persona and settings baked in.

<Callout type="key" title="Where Ollama fits">

Ollama is the right tool for **local development, prototyping, and single-machine/edge deployment** — it's the quickest way to try models and gives you an OpenAI API for free. It handles concurrent conversations fine for a handful of users, but for **high-throughput production serving** (hundreds of concurrent users) you want vLLM's PagedAttention and continuous batching. Rule of thumb: **prototype on Ollama, serve at scale on vLLM** — and because both speak the OpenAI API, switching is a one-line change.

</Callout>

## Recap

- **Ollama wraps llama.cpp** with a package-manager UX: `pull` models by name, automatic GPU detection, and a built-in server — zero config.
- The CLI (`pull` / `run` / `list` / `rm`) downloads ready-made **Q4_K_M** models; first run auto-offloads to whatever GPU you have.
- It serves both a **native REST API** and an **OpenAI-compatible** one on port 11434 — so code is interchangeable with vLLM and OpenAI.
- **Modelfiles** bake a base model + system prompt + parameters into a reusable named model (a `Dockerfile` for personas).
- Use Ollama for **local/prototype/edge**; move to **vLLM** for high-throughput production — switching is a one-line `base_url` change.
