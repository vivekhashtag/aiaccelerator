## The DSP Constraint Forces the Issue

On a GPU you're resource-rich — 19,456 CUDA cores plus Tensor Cores, and dropping FP32→FP16 just doubles throughput. On the PYNQ-Z2 you have **220 DSP blocks for every multiplication in the entire network**. That scarcity makes precision the dominant design decision — lower precision doesn't just save memory, it directly multiplies how many MACs fit:

## Binary Neural Networks: MACs Without Multipliers

A **Binary Neural Network (BNN)** constrains all weights and activations to `{−1, +1}`. The magic is that a multiply between two such values is just a **comparison** — and a whole dot product becomes bitwise operations:

<XnorPopcount caption="With w,x ∈ {−1,+1}, the product is +1 when signs agree and −1 when they differ — exactly XNOR of their sign bits. A full dot product is then 2·popcount(XNOR) − N: an XNOR followed by a bit count. No multiplier, so zero DSPs — it runs in LUTs, giving ~50–100× the parallelism of INT8." />

<Callout type="insight" title="Why this is the FPGA superpower">

A 256-element INT8 dot product would need 256 DSPs — more than the board *has*. The same dot product in binary takes ~50 LUT6 cells and **no DSPs at all**. That's why binary/ternary quantization is the signature FPGA technique: it converts the scarce, expensive resource (DSPs) into the abundant one (LUTs). The cost is accuracy — fine for MNIST-class problems (98%+), lossy for ImageNet (~60–65% vs 75%+) — so it's a deliberate trade, not a free lunch.

</Callout>

## FINN: Compiling Quantized Nets to Hardware

You don't hand-write all this. **FINN** (Fast Inference on FPGAs with Neural Networks, Xilinx Research) takes a quantized PyTorch model and generates a streaming-dataflow FPGA accelerator:

<FinnPipeline caption="A Brevitas-trained quantized model is exported to FINN-ONNX; the FINN compiler folds BatchNorm into weights, maps each layer to a custom streaming hardware block, and emits RTL; Vivado synthesises and places it; PYNQ runs it. The per-layer blocks (StreamingFCLayer, Thresholding) are wired by AXI-Stream FIFOs — the dataflow pattern from 8.2." />

FINN generates a custom streaming block per layer — `StreamingFCLayer` (pipelined fully-connected), `ConvolutionInputGenerator` (im2col patches), `Thresholding` (one operation that fuses ReLU + BatchNorm + quantize) — connected into the dataflow pipeline you saw in Lesson 8.2.

## Brevitas: Training the Quantized Model

**Brevitas** is the PyTorch extension that trains the quantization-aware model FINN consumes. Its layers mirror `torch.nn` but with quantization built in:

```python
import torch.nn as nn
import brevitas.nn as qnn
from brevitas.quant import Int8WeightPerTensorFloat, Int8ActPerTensorFloat

class QuantizedCNN(nn.Module):
    def __init__(self):
        super().__init__()
        self.conv1 = qnn.QuantConv2d(            # like nn.Conv2d, but quantized
            1, 16, 3, padding=1,
            weight_quant=Int8WeightPerTensorFloat,   # INT8 weights
            return_quant_tensor=True,                # pass quant info to the next layer
        )
        self.relu1 = qnn.QuantReLU(act_quant=Int8ActPerTensorFloat)  # INT8 activations
        self.fc = qnn.QuantLinear(16*28*28, 10, weight_quant=Int8WeightPerTensorFloat, bias=False)

    def forward(self, x):
        x = self.relu1(self.conv1(x))
        return self.fc(x.view(x.size(0), -1))
```

**Step by step:**

1. `qnn.QuantConv2d` / `QuantLinear` are drop-in quantized versions of the standard layers — you pass a `weight_quant` policy (here INT8 per-tensor, the granularity choice from Module 6).
2. `return_quant_tensor=True` carries the quantization scale/zero-point **downstream**, so the next layer knows the exact integer format — essential for FINN to generate correct hardware.
3. `QuantReLU` quantizes activations too, so *both* operands of every MAC are integers — exactly what the DSP/LUT hardware needs.

For a **binary** net you swap in `BinaryWeight` / `BinaryAct` quantizers — but a few training rules are non-negotiable:

<Callout type="key" title="Training binary networks that actually converge">

Four rules: **(1)** use the **straight-through estimator** (Module 6) so gradients flow through the binarize step; **(2)** **BatchNorm is mandatory** — it normalises the distribution before binarization, and without it binary nets simply don't train; **(3)** keep the **first and last layers in higher precision** (INT8/FP) — binarizing raw pixel inputs or the softmax outputs loses too much; **(4)** use **LR warm-up**, because binary gradients are noisier. Follow these and a binary net hits 98%+ on MNIST.

</Callout>

## Recap

- The PYNQ-Z2's **220 DSPs** are the multiplication budget for the whole network, so **precision is the dominant design choice** — lower precision packs more MACs per DSP.
- **Binary networks** turn a MAC into **XNOR + popcount** (`2·popcount − N`) — **zero DSPs**, running in LUTs, for ~50–100× the parallelism of INT8, at an accuracy cost.
- **FINN** compiles a quantized PyTorch model into a **streaming-dataflow** FPGA accelerator (per-layer blocks + AXI-Stream FIFOs); **Brevitas** trains that model with `QuantConv2d`/`QuantLinear`.
- Binary training needs the **STE, BatchNorm, higher-precision first/last layers, and LR warm-up** — then it's MNIST-grade accurate.
